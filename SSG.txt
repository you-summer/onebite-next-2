SSG 정적 사이트 생성
(Static Site Generation)
- SSR의 단점을 해결하는 사전 렌더링 방식
- 빌드 타임에 페이지를 미리 사전 렌더링해둠

유저 - 브라우저 - 넥스트 앱 서버

넥스트 앱을 우리가 npm run build명령어로 빌드할때 즉, 빌드타임(Build Time)에 미리 사전 렌더링을 진행해서 페이지를 미리 딱 한번만 생성을 해 둡니다

그리고 나서 다시는 더 이상 새롭게 페이지를 생성하지 않아요

그래서 빌드가 완료된 이후에 넥스트앱이 실제로 가동되었을때 이런식으로 브라우저가 접속요청을 보내게 되면 넥스트 서버는 빌드타임에 미리 만들어 두었던 페이지를 미리 사전 렌더링이 마친 페이지를 지체없이 매우 빠른 속도로 바로 응답하게 됨 

그럼으로써 사용자들은 매우 빠른 시간 안에 완성된 화면을 볼 수 있게 됨

그래서 SSG는 이런 방식으로 동작하기 때문에 이런 빌드타임에 일어나는 사전렌더링 과정에서 백엔드 서버에게 데이터를 불러와야 하는 과정이 추가로 필요하다고 하더라도 그 과정이 특정 상황으로 인해서 매우 오래 걸리게 된다고 하더라도 사용자의 경험에는 아무런 영향을 미치지 않습니다 

왜냐면 이런 모든 상황들은 서버가 가동되기 이전인 빌드 타임에만 일어나는 상황이기 때문에 이 빌드 타임이후에 발생하는 접속 요청들에는 굉장히 빠른 속도로 이미 만들어ㅜㄴ 페이지를 바로 응답할 수 있게 되기 때문

그리고 이 이후의 과정은 서버 사이드 렌더링 방식과 동일하게 진행이 됨
그래서 마찬가지로 똑같이 브라우저는 서버로부터 받은 HTML 페이지를 화면에 렌더링 하게 됨
그리고 이어서 하이드레이션을 위해서 Next 서버가 JS 번들을 후속으로 전달하면 브라우저에서 이를 실행해서 하이드레이션을 완료함으로써 상호작용이 가능한 완성된 페이지로까지 거듭나게 됨

SSG의 이 방식은 사전렌더링 과정에 굉장히 많은 시간이 소요되는 페이지라도 빌드 타임에 미리 다 만들어 둘 거기때문에 사용자의 요청에는 언제나 똑같이 매우 빠른속도로 응답할 수 있다는 굉장히 훌륭한 장점이 있다 라고 정리를 해볼 수 있음

단점
- 빌드 타임이후에는 다시는 페이지를 새롭게 사전 렌더링 하지 않기 때문에 페이지를 새롭게 생성하지 않기 때문에 우리가 언제 접속 요청을 보내게 되더라도 매번 똑같은 페이지만 응답함
그렇기 떄문에 결국에는 최신 데이터를 반영하기에는 많이 어려움
- (매번 똑같은 페이지만 응답함, 최신 데이터 반영은 어렵다) 

그래서 SSG 방식은 최신 데이터가 빠르게 반영되어야 하는 페이지들 보다는 데이터가 자주 업데이트 되지 않는 정적인 페이지들에 훨씬 적합한 사전렌더링 방식임


SSG 실습
getStaticProps 라는 함수는 getServerSideProps 함수와 동일하게 이런식으로 사전 렌더링과정에서 필요한 데이터를 불러온 다음에 그렇게 불러온 데이터를 props로써 컴포넌트에게 전달하는 역할을 해줄수도있음

그렇기 떄문에 함수의 반환값 역시 이전에 살펴봤었던 getServerSideProps함수와 동일하게 props라는 프로퍼티를 포함하는 객체를 내보내주도록 설정해줘야함

페이지 컴포넌트에서는 이전과 똑같이 getStaticProps 함수로부터 전달받은 데이터들을 props로 받아와서 자유롭게 컴포넌트 내부에서 사용해주면됨

대신에 이제는 props의 타입을 정의할 때 기존에 썼던 것처럼 InferGetServerSidePropsType 이런 타입을 쓰는 게 아니라 대신에 InferGetStaticPropsType 이라는 타입을 사용한 다음 제네릭으로는 typeof getStaticProps를 전달해주면됨
InferGetStaticPropsType<typeof getStaticProps>

이때 참고로 InferGetStaticPropsType이라는 타입의 역할은 앞서 InferGetServerSideProps타입의 역할과 똑같이 getStaticProps 함수의 반환값 타입을 자동으로 추론해서 props의 타입으로 설정해주는 역할을 함

useEffect를 사용해서 하면됨 그래서 state에 보관하면 됨 

 export default function Page() {
  const [books, setBooks] = useState<BookData[]>([]);

  const router = useRouter();
  const { q } = router.query;

  const fetchSearchResult = async () => {
    const data = await fetchBooks(q as string);
    setBooks(data);
  };

  useEffect(() => {
    if (q) {
      // 검색 결과를 불러오는 로직
      fetchSearchResult();
    }
  }, [q]);
  return (
    <div>
      {books.map((book) => {
        return <BookItem key={book.id} {...book} />;
      })}
    </div>
  );
}

그리고 그 말은 곧 id.tsx라는 파일은 url파라미터로 전달이 되는 id 값에 따라서 동적으로 여러개의 페이지를 렌더링 할수 잇다는 말과 똑같음 
그래서 id가 1번일때는 1번 도서의 페이지를 렌더링
2번일때는 2번 도서를
100번일때는 100번도서를 렌더링하도록 동적인 페이지로써 영어로는 다이나믹한 페이지로서 설정이 되어있다는거임 
그리고 그렇기 때문에 이러한 페이지를 Next.js 서버 측에서 빌드타임에 SSG방식으로 사전 렌더링하여 미리 생성해두기 위해서는 먼저 선수과정으로 이 페이지에 어떠한 URL파라미터들이 존재할 수 있는지, 그럼으로써 어떠한 경로들이 존재할 수 있는지 설정하는 과정이 반드시 필요하게 됨 
그렇지 않으면 어떤 경로들이 있을 수 있는지 모르니까 아예 사전 렌더링을 진행할수 조차 없음
그렇기 때문에 사전 렌더링 이전에 빌드타임에 어떤 경로들이 존재할 수 있는지 설정하는 작업이 꼭 필요하다고 생각하면 됨

이 과정에서 백엔드 서버로부터 현재 데이터베이스에 등록된 도서들이 아이디 목록을 받아온다던지 임의로 몇개 아이디를 설정한다던지 해서 URL파라미터로써 아이디의 값이 1번,2번,3번이 있을 수 있다 라고 설정을 해 주게 되면 그 다음 과정인 사전 렌더링 과정에서는 이러한 경로들에 해당하는 모든 페이지들을 각각 다 생성하게 됨 

그래서 앞선 과정에서 URL파라미터로 전달되는 값이 1,2,3번이 있을 수 있다라고 설정을 해 두었으니까 사전렌더링 과정에서도 book/1.html, book/2.html, book/3.html 이라는 3개의 문서를 사전 렌더링 하게 된다는거임 

그리고나서 빌드가 종료된 이후에 브라우저가 이렇게 book/1 페이지로 접속을 요청하게 되면 앞서 만들어 두었던 book/1.html 페이지를 전달해주는거임! 그리고 마찬가지로 book/2 페이지로 접속요청을 보내게 되면 book/2.html 페이지를 반환하는 방식으로 동작하게됨

그래서 결론적으로 Next앱에 북페이지 처럼 동적인 경로를 갖도록 설정이 된 페이지에 SSG를 적용하시려면 반드시 사전렌더링이 진행되기 전에 이 페이지에 존재할 수 있는 모든 경로들을 직접 설정하는 작업을 선수로 진행해줘야함 
이때 이 역할을 하는 함수가 오류메세지로 떴던 getStaticPaths라는 함수임
그래서 정리하자면 getStaticPaths라는 함수를 호출해서 현재 이 페이지에 존재할 수 있는 경로들을 먼저 설정한 다음에 그렇게 설정된 경로에 해당한 페이지들을 getStaticProps 함수를 일일이 한 번씩 다 호출해서 사전에 여러개의 페이지를 렌더링하는 방식으로 동작시켜야 된다 라고 이해하면 됨

[id].tsx 파일안에 getStaticProps 함수 위에 
export const getStaticPaths라는 함수를 화살표 함수로 선언함
이 함수에서 return으로 객체를 반환할건데 객체안에 paths라는 값으로 현재 이 페이지에 어떤 URL파라미터들이 존재할 수 있는지, 어떤 경로들이 존재할 수 있는지 배열로 반환하도록 설정해주면됨

임시로 현재 이 페이지에는 1,2,3번 아이디가 있을 수 있다 라고 설정하려면
path배열 안에 하나의 경로 아이템을 객체로 설정을 해줘야함
url파라미터를 의미하는 params라는 값으로 id : "1" 이런식으로 설정해주면됨

export const getStaticPaths = () => {
  return {
    paths: [
      { params: { id: "1" } }, 
      { params: { id: "2" } },
      { params: { id: "3" } },
    ],
    fallback: false, //존재하지 않는경로는 not found페이지
    // 대체 , 대비책, 보험
  };
};

결론적으로는 url파라미터로 id값이 1번,2번,3번이 들어올 수 있다는 거고 그렇게 되면 book/1, /2, /3이라는 페이지가 존재할 수 있다 라고 설정한 거라고 이해해주면 됨
이때 참고로 이 URL파라미터의 값은 반드시 문자열로만 명시해줘야됨!
그래야만 Next가 정상적으로 경로들을 읽어올수 있기 때문
프레임워크의 문법상 이 URL파라미터의 값들은 반드시 문자열로 설정을 해줘야한다!
fallback 하면 존재하지 않는 book/4같은 걸로 접속 요청을 하게 되면 
그때에는 어떻게 할건지 대비책을 설정하는 옵션이라고 생각하면 됨
fallback에는 총 3가지의 대비책 즉 3가지의 옵션이 존재함 
false는 not found page ㄱㄱ


-Fallback 옵션 설정하기

false : 404 Not Found 반환
blocking : 즉시 생성 (Like SSR)
true : 즉시 생성 + 페이지만 미리 반환

blocking
 - 없는 경로로 요청이 들어왔을때 해당하는 페이지를 SSR(서버사이드렌더링)방식처럼 넥스트 서버측에서 즉시 사전 렌더링을 거쳐서 생성해서 반환해 주게 됨
넥스트앱의 빌드타임에 book/1, book/2, book/3 페이지만 생성을 해 두었는데
이때 만약에 브라우저에서 book/100 이라는 존재하지 않는 경로로 요청을 보내게 되면 
우리가 설정한 fallback 의 값이 false였을때는 이렇게 404 not found 페이지를 리턴했지만 fallback 옵션이 blocking일때는 ssr 방식처럼 실시간으로 요청받은 페이지를 사전 렌더링해서 브라우저에게 반환해주는 방식으로 동작하게 됨
그렇기 때문에 blocking 옵션을 이용하면 빌드 타임때 사전에 생성해두지 않았던 페이지 까지 사용자에게 제공해줄수 있다는 장점이 있음

주의사항은?
존재하지 않았던 페이지를 SSR 방식으로 새롭게 생성할 때 백엔드 서버에게 추가적인 데이터를 요청해야 된다거나 해서 페이지의 생성시간, 사전 렌더링을 하는 시간이 길어지게 될 경우에는 이 시간동안 우리 브라우저에게 넥스트 서버가 아무것도 응답하지 않기 때문에 어쩔수 없이 로딩이 발생하게 됨

페이지의 크기에 따라서 오랜시간을 기다려야 할수도있음 ㅜㅜ

이거의 해결방법
fallback의 true
브라우저로부터 존재하지 않는 페이지를 요청 받았을때 일단 props가 없는 버전의 페이지를 빠르게 생성해서 브라우저에게 지체없이 반환해주게 됨
이때 porps라는건 페이지에 필요한 데이터를 계산하는 getStaticProps 의 return값들을 말함

그래서 props가 없는 페이지라고 하면 그냥 데이터가 없는 페이지다!
(props가 없는 페이지 : getStaticProps로 부터 받은 페이지가 없는 페이지)
데이터를 백엔드 서버로부터 불러오는 복잡하고 오래 걸리는 과정은 생략하고
그냥 컴포넌트가 렌더링하는 레이아웃 정도만 렌더링 해주기 위해서 props가 없는 페ㅣ지만 사전 렌더링을 해서 반환을 하게 됨

페이지에 필요한 데이터인 props만 따로 계산을 해서 계산이 완료되면 그때 브라우저에게 후속으로 또 보내주게 됨
데이터만 따로 나중에 보내준다는거임

브라우저의 입장에서는 데이터가 없는 버전의 페이지를 받아서 화면에 렌더링
이때 로딩바 같은걸 보여주면됨 그러다가 나중에 서버가 props의 계산을 마쳐서 props의 값만 따로 전달해주게 되면 그때는 데이터가 들어온거니까 데이터를 포함한 페이지를 화면에 뒤늦게 렌더링하게 된다는거임!
fallback을 true로 설정해주게되면 우리가 빌드타임에 생성해놓지 않은 페이지를 blocking처럼 제공할수있으면서도 동시에 사용자에게 긴 로딩시간 대신 데이터가 없는 버전의 페이지라도 일단 먼저 보여줄수있다는 장점이 존재함

props를 계산하는 getStaticProps 함수의 호출은 생략하고 바로 이 페이지 컴포넌트만 사전 렌더링해서 빠르게 브라우저에게 보내준다는 거임

그러고나서 넥스트 서버측에서 따로 getStaticProps함수를 실행해서 props로 활용될 데이터들을 계산해서 후속으로 브라우저에게 데이터만 따로 전달을 해준다는것
ui를 먼저 렌더링하고 데이터는 나중에 전달해준다~


ISR 증분 정적 재생성 (Incremental Static Regeneration)
- 증분 정적 재생성
단순히 SSG 방식으로 생성된 정적 페이지를 일정 시간을 주기로 다시 생성하는 기술!
어떤 페이지를 SSG 방식으로 동작하도록 설정해서 해당 페이지를 빌드타임에 정적으로 생성하도록 설정해두면 생성된 페이지는 이 시간 이후에는 다시는 생성이 되지 않기 때문에 이 페이지를 언제 요청하더라도 매번 그냥 원래 만들어두었던 똑같은 페이지만 계속 반환이 됨

속도는 빠르지만 최신 데이터를 반영하기에는 많이 어렵다는 단점이 있었음
ISR방식은 SSG 방식으로 빌드타임에 생성된 정적인 페이지에 음식처럼 유통기한을 설정해 둘수가 있음
그래서 유통기한이 지나기 전에는 매번 똑같은 페이지를 계속해서 응답해주다가
유통기한이 딱 끝나는 순간 이후부터는 넥스트 서버 측에서 다시 새롭게 해당 페이지를 정적으로 생성함으로써 그 이후에 발생하는 접속 요청부터는 새로운 페이지를 반환할수있도록 일정 시간을 주기로 페이지를 다시 생성하도록 설정할 수가 있음

타이머로 잰듯이 칼같이 업데이트하는 정도까지는 아님!
시간의 흐름이 왼쪽에서 오른쪽으로 흘러간다고 가정했을때
ssg방식대로 빌드타임에 빌드타임에 페이지를 한번 생성하게 되면서
isr적용해서 60초 이후에 페이지를 다시 생성하도록 설정해두면
60초전에는 기존의 페이지를 반환하다가
60초 이후에는 이때는 일단 원래 갖고있던 페이지를 반환하고 그리고 나서 서버에서 새로운 페이지를 다시 생성하게 됨 재생성을 하게됨 
이때 최신 데이터를 반영한 새로운 버전의 페이지가 새롭게 생성됨
그래서 그 이후부터 발생하는 요청에는 새롭게 생성된 버전의 페이지가 반환되면서 새로운 데이터가 반영된 페이지가 브라우저의 화면에까지 잘 렌더링이 됨
특정 시간 이후에 정적으로 생성된 페이지를 다시 새롭게 재생성하도록 설정해주는 ISR방식은 기본적으로는 이미 만들어져있는 페이지를 반환하기 때문에 굉장히 빠른속도로 브라우저에게 응답이 가능하다는 SSG 방식의 장점과 주기적으로 업데이트 해줄 수 있기 때문에 최신 데이터를 반영할수 있는 SSR 방식의 장점까지 함께 가지고 있는 강력한 렌더링 전략이다~

기존 SSG 방식의 장점 - 매우 빠른 속도로 응답 가능
기존 SSR 방식의 장점 - 최신 데이터 반영 가능

사용방법은 기존 getStaticProps의 return문 안의 props 객체 다음에 revalidate (재검증하다) 쓰고 몇초주기로 재검증할건지 해주면 됨!
  return {
    props: {
      allBooks,
      recoBooks,
    },
    revalidate: 3,
  };

ISR 방식의 추가적인 기능 - On Demand ISR 기능
ISR은 미리 생성해둔 정적 페이지를 응답하기 때문에 매우 빠른 속도로 동작한다는 SSG 방식의 장점과 더불어 특정 시간을 주기로 페이지를 다시 생성해서 최신 데이터까지 함께 반영시킬수 있다는 SSR방식의 장점까지 함께 갖는 강력한 사전렌더링 전략이다
SSR이나 SSG만 사용하기보다는 두 방식의 장점을 모두 다 가지고있는 ISR이라는 방식을 최대한 이용하는게 좋음

이런 ISR 방식을 적용하기에는 좀 어려운 페이지도 존재함
그래서 어떤 페이지들이 그런경우에 해당하냐면
시간과 관계없이 사용자의 행동에 따라 데이터가 업데이트 되는 페이지

예를들면 커뮤니티 사이트의 게시글 페이지~
시간과는 관계없이 게시글 수정이나 삭제 등의 사용자의 행동에 따라서 즉각적으로 업데이트가 필요함 
그렇기 때문에 기존의 ISR방식대로 60초마다 재검증하도록 해놨다면
60초가 되기 전에 게시글을 수정할수도 있는거임!

그래서 최신데이터를 즉각적으로 반영하기 어렵다는 문제점이 존재함 

60초 이후 재생성되도록 설정해뒀는데 24시간 이후에 게시글을 수정했다면
최신데이터에는 문제가 없지만 불필요하게 페이지를 다시 생성하게 되는 과정이 발생할거임

이 페이지는 60초마다 다시 생성되도록 설정이 되어있기 때문에 
수정이 되던 말던 그냥 60초마다 다시 생성하게 되는거임

실제로 다시 만들필요가 있나? no !
게시글의 수정은 하루뒤에나 되기 때문에 이런 상황에서는 불필요하게 페이지의 재생성이 발생하게 된다~

그래서 SSR로 처리하면안되나요?
SSR의 경우에는 브라우저가 요청할때마다 매번 새롭게 페이지를 사전 렌더링 하기 때문에
응답시간도 많이 느려지게 되고 동시에 접속자들이 많이 몰리게 될 때에는 서버의 부하까지 커져버림
되도록이면 정적인 페이지로써 처리해주는게 좋음

지금의 ISR방식으로는 많이 부족한게 현실임 
그래서 Next는 시간을 기반으로 페이지를 업데이트 시키는 기존의 ISR방식이 아닌 요청을 기반으로도 페이지를 업데이트 시킬 수 있는 새로운 방식의 ISR도 제공하고 잇음
요청 받을때마다 페이지를 다시 생성하는 ISR방식!
On-Demand ISR이라고 부름
요청이 들어올때마다~ 주문이 들어올때마다~ 라는 뜻
On Demand ISR 방식을 이용하면 시간을 기반으로 불필요하거나 너무 늦게 페이지를 재생성하는 과정들은 싹 제거하고 실제로 사용자가 이 게시글을 수정해서 페이지의 데이터 업데이트가 필요해졌을때 넥스트 서버에게 우리가 직접 페이지를 재생성하라는 요청인 페이지 리벨리데이트 요청을 보내서 페이지를 이때 다시 생성하도록 설정해줄수있음 

이 방식은 페이지의 업데이트를 우리가 직접 트리거링 할수 있음
온디맨드ISR방식의 새로운방식을 이용하면 최신데이터를 반영하면서도 정적 페이지로써 처리해줄수있기 때문에 이제부터는 온디맨드 ISR전략을 어떻게 설정할수있고 어떻게 페이지를 다시 생성하도록 요청할수있는지 살펴볼것
