SSG 정적 사이트 생성
(Static Site Generation)
- SSR의 단점을 해결하는 사전 렌더링 방식
- 빌드 타임에 페이지를 미리 사전 렌더링해둠

유저 - 브라우저 - 넥스트 앱 서버

넥스트 앱을 우리가 npm run build명령어로 빌드할때 즉, 빌드타임(Build Time)에 미리 사전 렌더링을 진행해서 페이지를 미리 딱 한번만 생성을 해 둡니다

그리고 나서 다시는 더 이상 새롭게 페이지를 생성하지 않아요

그래서 빌드가 완료된 이후에 넥스트앱이 실제로 가동되었을때 이런식으로 브라우저가 접속요청을 보내게 되면 넥스트 서버는 빌드타임에 미리 만들어 두었던 페이지를 미리 사전 렌더링이 마친 페이지를 지체없이 매우 빠른 속도로 바로 응답하게 됨 

그럼으로써 사용자들은 매우 빠른 시간 안에 완성된 화면을 볼 수 있게 됨

그래서 SSG는 이런 방식으로 동작하기 때문에 이런 빌드타임에 일어나는 사전렌더링 과정에서 백엔드 서버에게 데이터를 불러와야 하는 과정이 추가로 필요하다고 하더라도 그 과정이 특정 상황으로 인해서 매우 오래 걸리게 된다고 하더라도 사용자의 경험에는 아무런 영향을 미치지 않습니다 

왜냐면 이런 모든 상황들은 서버가 가동되기 이전인 빌드 타임에만 일어나는 상황이기 때문에 이 빌드 타임이후에 발생하는 접속 요청들에는 굉장히 빠른 속도로 이미 만들어ㅜㄴ 페이지를 바로 응답할 수 있게 되기 때문

그리고 이 이후의 과정은 서버 사이드 렌더링 방식과 동일하게 진행이 됨
그래서 마찬가지로 똑같이 브라우저는 서버로부터 받은 HTML 페이지를 화면에 렌더링 하게 됨
그리고 이어서 하이드레이션을 위해서 Next 서버가 JS 번들을 후속으로 전달하면 브라우저에서 이를 실행해서 하이드레이션을 완료함으로써 상호작용이 가능한 완성된 페이지로까지 거듭나게 됨

SSG의 이 방식은 사전렌더링 과정에 굉장히 많은 시간이 소요되는 페이지라도 빌드 타임에 미리 다 만들어 둘 거기때문에 사용자의 요청에는 언제나 똑같이 매우 빠른속도로 응답할 수 있다는 굉장히 훌륭한 장점이 있다 라고 정리를 해볼 수 있음

단점
- 빌드 타임이후에는 다시는 페이지를 새롭게 사전 렌더링 하지 않기 때문에 페이지를 새롭게 생성하지 않기 때문에 우리가 언제 접속 요청을 보내게 되더라도 매번 똑같은 페이지만 응답함
그렇기 떄문에 결국에는 최신 데이터를 반영하기에는 많이 어려움
- (매번 똑같은 페이지만 응답함, 최신 데이터 반영은 어렵다) 

그래서 SSG 방식은 최신 데이터가 빠르게 반영되어야 하는 페이지들 보다는 데이터가 자주 업데이트 되지 않는 정적인 페이지들에 훨씬 적합한 사전렌더링 방식임
